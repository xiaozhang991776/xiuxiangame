# 玩法优化方案 · 仙途·轮回诀

> 目标：移除冗余「总效率」指标及其计算逻辑，优化核心玩法反馈与理解成本，
> 在不改动任何数值平衡的前提下，解决新手体验阻塞与信息过载问题。
> 交付包含：① 删除模块清单 ② 玩法优化方案（改动前后对比）③ 潜在风险应对策略。

---

## 一、删除模块清单

| # | 位置 | 内容 | 类型 | 说明 |
|---|---|---|---|---|
| 1 | `index.html` · 修炼面板 `.cultivate-stats` | `<div class="cs-item cs-total">` 整块（含「总效率」标签与 `id="totalRate"`） | DOM 节点 | 面板内冗余速率显示 |
| 2 | `js/ui.js` · `renderCultivate()` | `document.getElementById('totalRate').textContent = detail.total.toFixed(2) + '/s';` | JS 赋值 | 对上节点的写入，删后无引用 |
| 3 | `js/cultivate.js` · `getRateDetail()` | 返回值中的 `total: base` 字段 | JS 计算字段 | 仅用于「总效率」显示，无其他消费方 |
| 4 | `css/style.css` | `.cs-item.cs-total {…}` 与 `.cs-total .cs-value {…}` 两条规则 | CSS 样式 | 随 DOM 节点一并移除 |

### ⚠️ 明确「不删」的项（避免奖励断层）

- **`player.stats.totalXiu`（累计修为）** —— 这是**完全不同的概念**（累计获得总量），
  驱动 `xiu_total` 类任务进度（`explore.js` / `save.js` 多处 `tickProgress('xiu_total', …)`）。
  误删会导致任务进度永久卡死，正是需求警示的「设计漏洞 / 奖励断层」。**完整保留**。
- **顶部资源栏 `topRate`（实时速率 `+x.x/s`）** —— 与「总效率」数值同源但展示位置不同，
  移除面板「总效率」后，玩家仍可在顶栏看到实时修炼速率，**信息无丢失**。
- **`getRateDetail()` 其余字段**（`base` 基础效率、`realmBonus`、`gongfaBonus`、`equipBonus`）——
  仍用于修炼面板的「基础效率 / 境界加成 / 功法加成 / 装备加成」明细展示，**全部保留**。

## 二、玩法优化方案（改动前后对比）

### 优化 A：移除冗余「总效率」指标（降复杂度）

| | 优化前 | 优化后 |
|---|---|---|
| 修炼面板数字 | 基础效率、境界加成%、功法加成%、装备加成%、**总效率**（5 项） | 基础效率、境界加成%、功法加成%、装备加成%（4 项） |
| 信息关系 | 「总效率」= 前 4 项相乘结果，同时又与顶栏 `topRate` 同源 → **双重冗余 + 叠乘易让新手误以为有额外加成** | 面板只展示「来源拆解」，最终速率统一看顶栏，概念唯一 |
| 理解成本 | 高（新手困惑「总效率」和「基础×加成」的关系） | 低（加成就是加成，结果看顶栏） |

**收益**：消除重复信息造成的认知负荷，让「修为从哪来」一目了然；无数值改动，零平衡风险。

### 优化 B：修炼面板新增核心循环引导（解体验阻塞）

| | 优化前 | 优化后 |
|---|---|---|
| 新手首屏 | 仅一句状态文案「气机流转，修为缓缓增长」+ 两个按钮，**无目标指引** | 状态文案下方新增一句引导：<br>「修炼聚气攒 **修为** → 修为满则点 **突破** 精进境界 → 境界越高，去 **斗法历练**、**秘境探宝** 收获越丰，循环往复，问道长生。」 |
| 玩家行为 | 易陷入「我该干嘛」的停滞（典型体验阻塞点） | 一眼看懂主循环：修炼 → 突破 → 战斗/探索 → 更强 → 再修炼 |

**实现**：`index.html` 新增 `.cultivate-guide` 段落；`css/style.css` 增 3 条样式（玉色左边框 + 半透明底，金色高亮关键词），视觉轻量不抢戏。

### 优化 C：核心循环反馈节奏（确认已就绪，未改数值）

| 环节 | 现状（保留） | 是否调整 |
|---|---|---|
| 突破就绪 | `.breakthrough-btn.ready` 已有 `breakReady` 脉冲动画（金/红辉光呼吸） | 保留，未动 |
| 修为进度条 | `.cultivation-bar-fill` 已有 `transition: width 0.4s` + 流光 shimmer | 保留，未动 |
| 手动修炼 | 「修炼」按钮点击有脉冲 + 「修为 +N / 连击×N」飘字 | 保留（上轮已加），未动 |

**说明**：反馈基础设施已具备，本轮仅做「精简 + 指引」，不叠加新动效，避免喧宾夺主与性能开销。

## 三、潜在风险与应对策略

| 风险 | 等级 | 触发条件 | 应对策略 |
|---|---|---|---|
| **误删 `totalXiu` 导致任务断层** | 🔴 高 | 将「总效率」与「累计修为」混淆 | 已严格区分：仅删「总效率」显示与 `getRateDetail.total`；`player.stats.totalXiu` 及全部 `tickProgress('xiu_total',…)` 调用**原样保留**。全局 grep 复核无残留误删。 |
| **移除后顶栏速率不显示** | 🟡 中 | 误以为「总效率」是唯一速率展示 | 顶栏 `topRate` 由 `SaveSystem.calcCultivateRate()` 独立计算，与面板「总效率」解耦，移除后面板外仍有实时速率，信息完整。 |
| **`getElementById('totalRate')` 空引用报错** | 🟡 中 | 只删 HTML 没删 JS 赋值 | 已同步删除 `ui.js` 赋值语句 + `cultivate.js` 返回字段 + `css` 样式，四处置零，无空引用。 |
| **数值平衡被破坏** | 🟢 低 | 改动了 `calcCultivateRate` 或 config | 本轮**零数值改动**：未碰 `calcCultivateRate()`、未碰 `GameConfig` 任何数值。修为产出、突破成本、战斗数值完全不变。 |
| **回归：修炼面板渲染异常** | 🟢 低 | 编辑 HTML 结构错位 | 已对 `cultivate.js` / `ui.js` 执行 `node -c` 语法检查通过；`detail.base` 等仍被使用的字段确认保留。 |
